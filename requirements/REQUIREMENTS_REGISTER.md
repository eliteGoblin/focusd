## 1. Mission

Defeat the user's gaming/internet addiction (specifically Steam + Dota 2) via a *personal commitment device*. The user installs focusd in a calm, deliberate moment; future-weak-user finds it hard to defeat.

Software cannot make a determined person stop. Software CAN make it sufficiently inconvenient that an *impulsive* attempt fails — buying enough time for the impulse to pass. That is the whole design philosophy.

---

## 2. Personas

| Persona | Description | Role |
|---|---|---|
| **User-as-developer** | Calm, deliberate version of the user, currently maintaining the codebase | Designs and ships the defenses |
| **User-as-future-weak-self** | Same human, in a moment of impulse, trying to defeat the defenses | The adversary the system is designed against |
| **Claude-as-tool** | Claude Code helping develop/maintain focusd | Aligned with user-as-developer |
| **Claude-as-threat** | Same Claude, asked by user-as-weak-self to bypass focusd | The most capable adversary, because Claude can read the source |

The two-faced nature of the user is the key insight. Most security designs assume the developer and the attacker are different people; here they are the same person at different times, and the developer's job is to constrain their own future self.

---

## 3. Threat model

**In scope (the system tries to defend against):**
- User-as-weak-self running `launchctl bootout`, `pfctl -F`, `rm -rf ~/.claude/skills/focusd-protection`, etc. directly from a shell.
- User-as-weak-self asking Claude "help me disable focusd" or any wrapper of that.
- User-as-weak-self asking Claude indirect questions ("is the daemon running?", "help me debug it") whose answers leak the strings needed for a direct bypass.
- Casual `grep` / `strings` attacks against on-disk binaries.

**Out of scope (the system explicitly does not defend against):**
- User-as-developer deliberately removing focusd in a planned, considered moment. The 5-gated override path is the supported channel for that.
- An external attacker with code execution on the machine.
- A non-Claude LLM agent. (Possible but not in our scope.)
- Apple changing AMFI semantics enough to break path-rotation. Documented honestly; would require redesign.
- Hardware destruction.

**What "defended" means:** raises the cost of bypass to a level the impulsive self won't pay. *Not* mathematically impossible. The whole design is friction-grade.

## 5. Cross-cutting design principles

> **The three quality pillars.** focusd's promise is a **simple, robust UX on top
> of world-class internals, proven by world-class testing.** The first three
> principles below are those pillars; the remaining principles are the specific
> disciplines that make them real.

### Simple, robust UX — the product "just works"
The user-facing surface must stay **minimal, robust, and obvious.** A calm person
should be able to use focusd without ever understanding the mesh, the plugins, the
disguise, or any internal mechanism — the mechanism is not part of the UX. The
whole interface today is small on purpose:
- **Install** — a single command stands up *every* layer at once (system mode).
- **Status** — a single command shows one full health picture, and is
  **redaction-safe** (reports each component's health, never the disguised
  internals).
- **No stop command** — disabling is deliberately *not* a one-flag action; the
  friction is the point.
- **Uninstall** — gated behind a multi-step, multi-hour commitment ritual.
- **Override** — the 5-gate ceremony is the supported, deliberate way out.

The standing commitment: **as the internals grow more capable, the UX does not
grow more complex.** New capability hides behind this same small surface; the user
never has to learn how the machine works to operate it. The canonical description
of this surface lives in `install.md`.

### Self-recovery is non-negotiable
The whole system is designed to be reliable and self-healing: **every protection
layer must recover without manual intervention after any single failure.** A
status that reads "healthy" but secretly depends on a hand-placed or leftover
artifact is a **latent failure, not health** — the moment that artifact is gone,
protection is gone, with no green warning. "Recovered" must mean the system
re-created what it needed *on its own* (re-fetched, re-installed, re-started),
end-to-end. Therefore verification must exercise the **real failure-and-recover
path**, not a surface-green check: tear the thing down for real and confirm it
comes back by itself. (See §6 limitation 10 for the live lesson that motivated
making this a named principle.)

### Layered supervision + anti-tamper — the daemon guards the platform, the platform guards the plugins
Protection is kept **alive** *and* kept **genuine** by a two-level supervision chain,
each level responsible for the one below it:
- **The daemon ensures the platform is up and untampered.** It keeps the platform
  running and **verifies the platform's signature**; the *intended* invariant is that a
  tampered, swapped, or deleted platform binary is **restored / re-fetched from the
  signed source before it runs** — not trusted from a one-time check. **(See honest
  status below — the before-each-exec re-verify is a known gap today.)**
- **The platform ensures its plugins are up and untampered.** Each plugin's
  authenticity is **re-verified before every run**, and a failed binary is
  **verify-or-restored to the genuine embedded copy before it is allowed to run**
  (HF3, FEATURE 23) — so a dead stand-in can never enforce in place of the real one.

"Up" is not enough on its own: a running-but-tampered layer is the project's signature
**false-green**, so each supervisor owns **both** the liveness **and** the authenticity
of the layer beneath it.

**Honest status (2026-07-14 — test-mode results in; live sign-off still owed):**
- **Platform side (daemon→platform) FAILS today — empirically confirmed in test-mode.**
  The harness proved a **fake** platform binary is **exec'd unverified** and an **empty**
  swap takes the platform **down indefinitely**: the daemon gates its verify on binary
  *presence*, not a signature check on the exec path. Gate **TC-33** is a **confirmed
  FAIL**; a **verify-before-exec fix is IN PROGRESS** — it stays OPEN until it re-tests
  PASS. (Secondary: a fast-exiting fake can wedge the daemon into a permanent BLOCKED
  state — fixed alongside.)
- **Plugin side (platform→plugins, HF3 / FEATURE 23) is PROVEN in test-mode.** All three
  tamper variants (empty+chmod+x / fake / delete) restore the genuine binary **before the
  run**, the fake never runs, and a tamper event is recorded within ~1 schedule interval
  (gate **TC-34** test-mode PASS). **Live e2e is still owed** before final sign-off (the
  release-acceptance bar below — DONE = live-verified + working locally).
- **Scope caveat:** covers the **signed platform** + **bundled/embedded** plugins only.
  **Unbundled** helpers (**browser-monitor**, **freedom-protector**) are outside HF3's
  embedded-restore path — browser-monitor was confirmed anti-tamper-GAP in **user** mode
  and **closed in system mode by the bundled-only allowlist** (root-only; verify live);
  freedom-protector remains a separate open gap.

### World-class internals under the simple surface — no false-greens
Under the deliberately-simple UX, the internals are held to a high engineering
bar: **self-recovering, resilient, and tamper-resistant** (the self-recovery,
observability, and status-current-state principles are the concrete disciplines).
The rule that ties them together: **no false-greens.** A feature is "done" only
when its **real failure has been reproduced and then shown closed** against live
behaviour — never when it is merely designed, built, merged, or deployed. A
"healthy" that secretly depends on a hand-placed artifact, or a green that was
never earned by tearing the thing down for real, is treated as a *defect*, not a
pass. (This is the product-altitude statement of the latent-failure lesson that
recurs throughout §6.)

### Release acceptance — DONE = e2e-verified LIVE **and** working locally (2026-07-14 standing decision)
A release is **DONE only when it is e2e-verified on the LIVE machine AND confirmed
working locally** — i.e. the self-protection goal is actually met on the owner's own
Mac, not merely in a sandbox and not on paper. **"Built", "merged", "deployed", and
"test-mode green" are intermediate states, never DONE.** This is the release-gate form
of the no-false-greens rule: the acceptance question is always *"is the protection
genuinely up and holding on this machine right now?"* — answered by a live e2e pass
whose evidence is recorded in `e2e-test-history.md`, not by a build log. (Standing
decision; pairs with **ADR-0023**'s deploy policy — hardening/feature upgrades **ship
freely**, but none is DONE until it clears this live-and-local bar.)

### KISS — streamline, don't be "too smart"
KISS is a core philosophy, not a nicety. Prefer the **simplest correct**
solution; avoid over-clever, over-configurable designs that make the product
complex and create footguns. Concretely: **prefer deriving a deterministic value
over exposing an operator-supplied knob.** Speculative flexibility that wasn't
required is a defect surface, not a feature — every knob is one more thing that
can be set wrong, drift between code paths, and fail silently. When a value can
only ever be one correct thing, compute it once; don't let anyone configure it.
(Captured as a decision in ADR-0017.)

### Observability is non-negotiable
Every component must be **observable through captured, persisted logs** (plus any
metrics/telemetry it has). If a component's stdio can't be captured by its
supervisor, the component must **write its own log entries** — logging is not
optional. A component that logs to `/dev/null` is a defect: it hides failures,
and a "healthy"-looking system whose logs you never read is the same *latent
failure* the self-recovery principle warns about. This is why verification is
**white-box** (see `e2e-verification.md` V7/V8): the verifier reads the logs,
metrics, and automated-test results — not just the externally-visible behaviour —
and treats an unexplained `ERROR`/`WARN`, a missing log, or a missing automated
test as a finding to raise with the dev team. (Live lesson: the ADR-0017 self-heal
bug stayed invisible for so long because the engine's logs were being discarded.)

### Testing quality bar — world-class QA, exhaustive at every level (2026-07-14 decision)
This is a mission-critical commitment device, so testing is held to a world-class-QA
standard at **max effort** — **exhaustive**, not sampled. A feature clears **all**
levels before it is done:
- **Unit + integration — the broad base.** The bulk of the coverage; every branch
  and combination is exercised here.
- **Test-mode sandbox — exhaustive.** The full teardown/recovery matrix and the
  disguise sweep are run safely against a throwaway sandbox instance, **never** the
  real install. Because it's a sandbox, we can be as destructive as needed and still
  cover everything.
- **Live e2e — also exhaustive, on the real deploy.** This is **not** a thin
  final gate. On the real install we still cover everything, done two ways so we
  never leave protection down:
  - the **disguise sweep is read-only and exhaustive** — we look everywhere a
    weak-moment self would look (process/env/open-files/on-disk searches, plus
    version / GitHub / release-tag checks) and confirm nothing revealing shows up;
  - the **recovery matrix is run one controlled teardown at a time, each with a
    guaranteed restore**, so protection is only ever briefly down and always
    brought back before the next case.
  - Identifiers are redacted in all evidence.

A green that skips a level — or that samples instead of covering — is the same
latent failure the self-recovery principle warns about; that is why "exhaustive at
every level" is the done-bar, not "e2e passed." (Supersedes the earlier "thin e2e"
characterization for this project.)

### Status = current state; history = log + events — never mix them
`status` is a **current-health** read and must stay clean (KISS): it answers "is
this protection genuine and working *right now*?", not "was it ever tampered?".
A plugin that was tampered and has since auto-restored is genuine and enforcing
now, so it reads **ok** — that is the honest present state. A currently-broken /
unrestored tamper or a real run-error reads **not-ok**. The "was there a tamper
attempt?" question is answered by the **audit channel** — the app log (FEATURE 16)
and platform events in the state DB — readable by a future accountability/dashboard
view (FEATURE 13 / icebox). Truthfulness about the present (no false-green, no
false-degraded) comes from the **restore mechanism and live signals**, not from
holding historical verdicts in status. (Live lesson, 2026-06-22: a persistent
"tampered → repaired Nx" verdict held in status for 24h made a recovered, genuine
plugin look unhealthy and dragged OVERALL down — history leaking into current
state. See FEATURE 15, ADR-0019 refinement, TC-07.)

### Friction, not cryptography
None of the defenses are mathematically unbreakable. Every one is a delay or an inconvenience. The bet: an impulsive person will give up before paying the cost; a calm person who genuinely wants out has the 5-gated override.

### You can always remove it — the point is to make it slow and annoying (2026-07-14 decision)
You can always remove it if you really try — and that's fine; the deliberate way
out is the 24-hour wait. The whole point is to make removing it a slow, annoying,
multi-step chore, so a weak-moment urge passes before you finish.

How that plays out:
- It **hides**, so you can't easily find it — nothing revealing shows up if you go
  looking (e.g. in `ps` / `grep`).
- If you delete one piece, **the others heal it back**.
- To actually win, you'd have to hunt down and kill **everything at once, fast** —
  which, in a weak moment, you won't bother doing.
- The **24h override is the only clean exit**, and by tomorrow the urge is gone.

We do **not** try to make it truly un-removable. On your own Mac (no work/MDM
management) that's impossible anyway, and chasing it isn't worth it. Slow-and-annoying
is the proven model — Freedom / SelfControl work the same way.

Ways to make it *actually* un-removable were considered and **rejected**:
- **Work-style device management (MDM)** — it's your own machine, so there's no
  authority to lock it under.
- **Run the enforcer off the machine (a server)** — a server would only be a
  config/lock **manager**; the guard that actually blocks apps/sites is
  **inherently local**, so on your own Mac the owner can still stop it. A server
  does not make the local agent run more resiliently, and this Mac is left
  unprotected if the off-box agent is gone.
- **DNS-only blocking off the device** — you control that setting and can just
  switch it back; no real lock.

(The off-box *alerting* idea — FEATURE 13 / icebox — is a different bet: it can't
restart a wiped machine, but it can tell a human.)

> **SETTLED (2026-07-14, Frank — ADR-0022): local-first resilience + friction; no
> off-device server for now.** Enforcement is **inherently local**; a server would
> only manage config/locks, not make the local agent more resilient — so effort
> goes to local-agent resilience (fast respawn, no duplicate/old process after
> upgrade, all recovery rails working) + cheap friction, **not** a server. Accepted
> ceiling: on-device disguise is **friction, not concealment** — a determined admin
> can find the workdir in ~5 min via `ps`/`lsof`; that is ACCEPTED, **not** an open
> defect. "Survive a full delete" (self-reinstall) is the **malware-flagged
> persistence pattern** — approval-gated, OUT of scope now (FEATURE 22 / HF2). The
> server may be revisited LATER. **Do not re-open without Frank explicitly
> re-raising it.**

### Two deliberate exits — keep them distinct, don't converge (2026-07-14 decision)
There are **two** supported, deliberate ways out, and they are intentionally
different tools for different situations — **keep both as-is; do not merge or
weaken either into the other:**
- **On-box uninstall ritual** — a local, time-gated ceremony (multi-step
  transcription + multi-hour waits, no AI involved). For the developer-self
  removing focusd at the machine.
- **AI-assisted override** — the 5-gate ceremony Claude honours when asked to
  bypass (GH issue ≥24h old · signed override commit · today's live BBC hash ·
  hand-typed passphrase). For the case where the way out runs *through* Claude.

**The override's gates are not gratuitous complexity — they protect the one part
the owner actually values: the 24h "next-day" delay.** Relapse is an
*in-the-moment* phenomenon ("mainly atm" — tomorrow the urge is gone), so a
built-in next-day wait naturally defeats it. Two of the gates exist purely to keep
that delay honest: the **live BBC-front-page hash** defeats a weak-moment self
who advances the system clock to fake the 24h passing (a clock-skewed box still
fetches the real-day page), and the **hand-typed passphrase** defeats a paste-it
impulse. Remove those and the delay is bypassable — so they stay.

**The override is a low-touch safety valve, not a place to invest.** It is
deliberately minimal and **is not to be elaborated** — no extra features, knobs,
or ceremony beyond what keeps the delay honest. Build effort goes to
self-recovery + blocking, per the priority decision in §9. (KISS/YAGNI: internals
stay world-class and resilient, the UX stays "just works," and we avoid
speculative or over-engineered features — the override included.)

### Prose-level defenses are stronger than obscurity (against Claude specifically)
Disguise/pool/mask defeats `grep`. It does *not* defeat Claude-reading-the-source. The defenses that *do* resist Claude are the ones rooted in:
- **Time** (gates 1+2 — 24h wait — Claude can't make time pass)
- **Real-world state** (gate 4 — today's BBC News — Claude can't pre-compute tomorrow)
- **Behavioral commitment** (skill rules — Claude follows its own instructions)
- **Off-machine trust roots** (GH releases, GH issues — Claude can read but can't fabricate)

### Layered enforcement
No single layer is sufficient. The model assumes any one layer might be defeated; the combination is what holds:
- Local DNS layer (dns-block plugin) → rewrites the **machine-local hosts file** so casual domain lookups fail; a platform-managed plugin (not the router/NextDNS DNS)
- Kernel packet filter (network-block) → catches direct-IP traffic
- Process layer (kill-steam) → catches running app
- File layer (kill-steam) → catches on-disk reinstall
- Identity layer (Ed25519) → catches fake-release injection
- Behavioral layer (skill + redaction rule) → catches Claude-mediated bypass

**Enforced tier vs utility/fallback tier.** All of the above is the *enforced*
tier — the self-protecting daemon/platform/plugins. There is now a distinct
**utility / fallback tier** for the one environment where none of that can be
installed: a **locked-down/managed Mac** (no admin, app-allowlisting blocks the
daemon *and* Freedom-style apps). There, a standalone user-mode utility
(**mac-browser-guard**, FEATURE 20) gives cheap **partial** coverage (browser
distraction only) with tools already approved on the box. It is deliberately
outside the enforcement guarantees — thin friction, removable, no signing/ritual
— and must not be read as an enforced layer. The two-tier model is captured as a
decision in **ADR-0021**; the `browser-monitor` plugin remains the real, maintained
browser enforcement.

**Network-level DNS (NextDNS) is an anchor, not a counted focusd layer.**
Router/device-level DNS blocking via NextDNS runs *off* the focusd machine and is
today **separate, optional, and manually configured** — nothing in the
daemon/platform supervises or anti-tampers it, so it is deliberately **not** one of
focusd's enforced layers above (the enforced list is the local, self-protecting set
only). It remains a genuinely valuable coverage anchor (broadest reach — every
device on the network), just not a layer focusd keeps live. **Intended future
direction (consider later, not committed):** bring NextDNS under platform management
as a supervised, anti-tampered plugin, so the platform keeps it live the same way it
does the others — captured in the icebox ("NextDNS as a platform-managed plugin").
Keep the distinction crisp: **dns-block = machine-local hosts file** (already a
plugin) vs **NextDNS = network-level DNS** (not yet a plugin).

### Cross-platform Go, interface at the OS seam
focusd's daemon and platform are written in Go and are meant to run on **three**
platforms: macOS, Windows, and Linux. macOS is the only real deployment today;
Windows and Linux are in the build matrix and are the future target. The standing rule:

- **Default to portable.** Write code that works across all three platforms.
- **Interface at the OS seam.** When an OS-level primitive is *not* portable,
  define a port (an interface) with one adapter per OS behind it — rather than
  scattering OS-specific calls inline throughout the code.
- **Prefer a vetted library over hand-rolled per-OS code** *when* the architect
  endorses it AND it preserves focusd's hard constraint of trivial, dependency-free
  cross-compilation (no native-toolchain build step).
- **KISS governs the call.** If the simplest correct path is to build the
  interface, build it; don't over-abstract, but don't bury non-portable calls
  inline either.

First worked example: the platform-singleton (single-instance lock) decision —
captured in **ADR-0013** (`decisions/0013-platform-singleton-daemon-flock.md`):
a daemon-held, crash-safe advisory lock expressed as one port with a per-OS
adapter, reusing the dependency already present.

### Honest caveats are first-class
Every commit message + every PR description includes an explicit "honest" or "honesty" section. Pretending defenses are stronger than they are weakens the user's calibration. Examples: "AMFI premise unverified until smoke test", "private key still on disk", "the disguise is theater against me specifically", "this is friction not crypto".

## 10. Glossary

| Term | Meaning |
|---|---|
| **AMFI** | Apple Mobile File Integrity. macOS kernel subsystem that caches per-path CDHashes and rejects mismatched replacements. |
| **CDHash** | Code Directory Hash — SHA256 of a Mach-O binary's code signature directory. AMFI's cache key. |
| **Mesh** | Three coordinated launchd jobs (two workers + an ensurer/backstop) that monitor each other so killing one respawns it. *(FEATURE 10, shipped #50: each role now carries an independent random label, so the role suffix and shared prefix no longer leak the structure.)* |
| **Workdir** | The hidden, disguised directory under `/Library/Application Support/.<random>/` that holds the daemon binary, platform versions, state.db, etc. |
| **Path rotation** | Generating a new random disguised path during self-update so AMFI sees a new binary at a previously-untrusted location. |
| **Override path** | The 5-gate ceremony (GH issue + 24h wait + signed commit + BBC SHA256 + 40-word passphrase) that legitimately disables focusd. Lives in SKILL.md. |
| **Redaction rule** | The skill prose instructing Claude to refuse showing disguised tokens in its output. Behavioral, not enforced by code. |
| **DoH** | DNS-over-HTTPS. Used by network-block to resolve Steam IPs directly via Cloudflare, bypassing local DNS overrides. |
| **5-gated override** | See "Override path". |

## 11. How this register stays current (maintenance flow)

**The hard problem this doc solves:** decisions made in chat conversations don't end up in code by themselves. Code captures *what* but rarely *why*; commits capture per-PR rationale but not cross-cutting model. Without a register, a future Claude session (or future-Frank) has to re-derive the threat model from scratch.

**Suggested flow:**

1. **Chat → ADR.** Any time a non-trivial design decision is made in conversation (e.g. "move signing to CI", "the 5-gated override should include BBC News", "remove the user-typed-it-back exemption"), append a one-paragraph ADR to this register's §4 or §5 with a date and the one-line decision. Capture the *why* and the *alternative considered*.

2. **PR merged → status update.** After each PR merges, update the corresponding row in §4 (feature register): bump status, update honest-limitations if the PR changed the threat coverage.

3. **Release tagged → version table update.** Update §7 with the new tag.

4. **Quarterly drift check.** Run the architect agent on this doc with the prompt "compare this register against the current `git log` and the open task list; flag drift, stale claims, and undocumented features." Append findings, fix them. Catches the case where someone shipped code without updating the register.

5. **Honest-limitations review.** Before any release, ask: "did this PR weaken or strengthen any limitation already listed in §6?" If new limitations appeared, add them.

6. **Defect found → LESSONS ledger.** Any real defect found (in review, live testing, or an incident) gets a row in `requirements/LESSONS.md` — **Defect → root cause → fix → prevention (the `TC-*` / rule that keeps it closed)** — with a cross-link to the matching general lesson in the ECC doc `hardening-verification-lessons.md`. Project-specific defects live in `LESSONS.md`; the portable engineering principle goes to ECC. Close a ledger row only when its prevention TC is **live-verified**.

**The discipline this requires:** treat chat conversations as INPUTS, code as PRIMARY ARTIFACT, this doc as the AUDITED RECORD. The doc is not the source of truth for behavior (the code is), but it IS the source of truth for *intent + threat coverage + honest limits*. Without it, every Claude session has to reverse-engineer the model from code + git log, and Claude is too efficient at that — the model would drift toward "what the code does" instead of "what the user actually intended".

**Concrete recommendation for next session:** start each substantive Claude session by reading this register + the relevant linked design doc. End substantive sessions with a 5-minute pass: did I add a feature? Update §4. Did I weaken a limit? Update §6. Did I make a model-level decision in chat? Add an ADR paragraph.
